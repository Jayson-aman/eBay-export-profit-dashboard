# Terapeak・落札相場データ — 仕様書

本ドキュメントは、eBay 輸出向け利益ツールにおいて **「実際にいくらで売れたか」** を扱うためのデータソース設計と運用ルールを定義する。

---

## 1. 背景と制約（公式 API）

| 手段 | 取得できるもの | 2026 年時点の制約 |
|------|----------------|-------------------|
| **Buy Browse API**（本ツール標準） | **現在出品中**の価格・件数 | キーワード検索可。落札済み一覧は返さない。 |
| **Finding API** `findCompletedItems` | 過去の「販売済み」検索 | **廃止方向**（コミュニティ上、代替が不明確な報告あり）。新規連携は推奨しない。 |
| **Marketplace Insights 等** | カテゴリ別の市場インサイト | **パートナー承認・契約**が必要なことが多い。汎用アプリでは未採用。 |
| **Trading API `GetMyeBaySelling`** | **自分の**売上・出品 | 他人の落札相場には使えない。 |
| **Terapeak（Seller Hub 内）** | キーワード別の **平均・中央値・販売件数** 等 | **UI 上のリサーチ**。公式の「キーワード検索用落札一括 REST」は一般向けに公開されていない前提で設計する。 |

**結論:** 本ツールは **Browse API＝現在出品の相場** と **手動・CSV の落札サンプル** を併用し、ユーザーが判断材料を揃えられるようにする。

---

## 2. データソースの役割分担

### 2.1 Browse API（自動）

- **用途:** いま競合が **何ドルに付けているか** の分布（速報性が高い）。
- **限界:** 高値で棚に残っているだけの出品が混ざる。落札に至った価格ではない。

### 2.2 Terapeak（手動メモ / スクリーンショット補助）

- **用途:** **実際に動いた価格帯**（平均・中央値・販売数）の目安。
- **運用:** Seller Hub → Research → Terapeak でキーワード・カテゴリ・期間を指定。表示された **平均・中央値・件数** をメモするか、下記 CSV に **代表的な数件** として転記する。
- **注意:** Terapeak の画面仕様は eBay 側で変更されうる。数値は **取得日時** を必ず記録する。

### 2.3 手動 CSV（本ツールの「落札モード」）

- **用途:** eBay サイトの **Sold フィルタ** で検索した結果、**落札価格が分かる行** をユーザーが転記したデータを統計処理する。
- **利点:** API に依存しない。**実売負の分布** に近い。
- **限界:** 入力の手間・抜け漏れ・主観的な商品選別バイアスがある。

### 2.4 第三者データ（任意・非推奨をデフォルト）

- 有料の相場 API、スクレイピングサービス等は **利用規約・著作権・eBay ポリシー** を各自確認のうえ利用。本リポジトリのデフォルト実装には含めない。

## 3. CSV スキーマ（`sold_price_csv.py` 準拠）

### 3.1 必須列（いずれかの列名で可 — 正規化時に吸収）

| 論理名 | 受け入れる列名の例 |
|--------|-------------------|
| 落札価格（数値） | `sold_price_usd`, `sold_price`, `price_usd`, `price`, `落札価格`, `total_usd` |

金額は **商品＋送料を分けたい場合** は次のオプション列を使う（後述）。

### 3.2 任意列

| 論理名 | 列名の例 | 説明 |
|--------|----------|------|
| 通貨 | `currency` | `USD`（省略時は USD 扱い。GBP/EUR 等は概算レートで USD 換算）。 |
| 商品タイトル | `title`, `product_name`, `商品名` | 一覧表示・トレース用。 |
| 落札日 | `sold_date` | `YYYY-MM-DD` 推奨。期間フィルタの将来拡張用。 |
| 商品価格のみ | `item_price` | 送料と分けたい場合。 |
| 送料（バイヤー負担） | `shipping_usd`, `shipping` | `total_usd` が無いとき `item_price + shipping` で合算可。 |
| 合計（落札＋送料） | `total_usd` | 最優先で「分析に使う 1 行あたりの USD」として採用。 |
| メモ | `notes`, `memo` | 例: Terapeak 画面のスクリーンショット番号。 |
| データ取得元 | `source` | 例: `terapeak`, `ebay_sold_search`, `manual` |

### 3.3 1 行あたりの「分析用 USD 金額」の決定ルール（優先順）

1. `total_usd` が正の数 → その値  
2. それ以外で `sold_price_usd` / `sold_price` / `price` が正 → その値（通貨換算）  
3. `item_price` と `shipping` の両方が取れる → 合算  
4. 上記いずれも不可 → **行スキップ**（エラー行としてカウント可能にする）

### 3.4 品質・監査

- **最低サンプル数:** 統計の推奨は **5 件以上**（外れ値除去後も 5 件未満の場合は警告を表示）。
- **取得日:** CSV 先頭行のコメントや `export_date` 列（将来）でよい。
- **版管理:** ファイル名例 `sold_kutani_vase_20260422.csv`

## 4. 統計ロジック（Browse API と揃える）

- 外れ値除去: **IQR 1.5 倍ルール**（サンプル数 8 件以上のとき）。
- 推奨売値の 4 段階:  
  - 速売 ≒ P25  
  - バランス ≒ 中央値 × 0.92  
  - 標準 ≒ 中央値  
  - 強気 ≒ P75  

※落札データでは「現在出品」と比べ **中央値の解釈が実売負に近い** ため、**最終売値は落札 CSV を優先する**運用を推奨（ダッシュボードで選択可能）。

---

## 5. Browse と落札 CSV の「見え方」

| 指標 | Browse API | 落札 CSV |
|------|------------|----------|
| 意味 | いまの出品価格の分布 | 入力された「実売負に近い」価格の分布 |
| 速報性 | 高 | 入力のタイミングに依存 |
| バイアス | 売れ残り高値など | サンプル選択バイアス |

**推奨ワークフロー**

1. Terapeak / Sold 検索で **カテゴリとキーワード** を固める。  
2. **Browse API** で現在の付け値レンジを把握。  
3. **落札 CSV**（数件でも）で実際の落札帯を記録。  
4. 利益計算の `sell_price_usd` は **CSV 由来の推奨値** を優先し、Browse は上限・下限の感覚として使う。

## 6. 将来拡張（オプション）

- eBay が **落札検索用の正式 REST** を公開した場合、`sold_price_csv.py` と同じ出力形式にマッピングするアダプタを追加する。  
- `sold_date` を使った **90 日加重中央値** 等。  
- Marketplace Insights の **契約済みキー** を環境変数で渡すアダプタ（別モジュール）。
